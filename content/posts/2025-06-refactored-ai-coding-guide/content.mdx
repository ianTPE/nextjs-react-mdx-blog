# 當你的大腦卡住時：AI 編程工具的真實使用體驗

> 「aider 是人想的比較多，codex 是人要去理解，claude 是沒有辦法去想和理解的時候，讓 claude 先出結果，再事後理解。」—— 一位開發者的真實感悟

## 序：從一個下午的掙扎說起

下午三點，我盯著螢幕上的程式碼發呆。一個看似簡單的需求——「將這個 React Class Component 重構成 Hooks」——卻讓我陷入了思維泥潭。不是不會寫，而是有太多種寫法，每種都有利弊，我在各種方案間來回糾結。

這時我打開了 Claude Code：

```bash
claude
> 把這個複雜的 Class Component 重構成 Hooks，保持所有功能不變
```

15 秒後，一個完整的解決方案出現了。我看著生成的代碼，突然明白了——原來可以用 `useReducer` 來管理這些複雜狀態。

**這就是 AI 編程工具的真正價值：不是替你寫代碼，而是在你思維卡住時，給你一個可能的出路。**

## 第一章：認知負擔與工具選擇

### 人腦的三種狀態

在實際開發中，我發現自己經常處於三種認知狀態：

**1. 清醒狀態：「我知道要做什麼」**

早上剛開始工作，需求明確，思路清晰。這時我會用 Aider：

```bash
aider src/utils/validation.js
> 添加 email 和 phone 的驗證函數，使用正則表達式
```

Aider 就像一個聽話的助手，精確執行我的想法。我不需要它有太多創意，只要能準確理解並實現我的意圖就好。

**2. 迷糊狀態：「我大概知道，但不確定」**

下午，遇到一個錯誤訊息：`TypeError: Cannot read property 'map' of undefined`

```bash
codex --attach error-screenshot.png
> 解釋這個錯誤並提供修復方案
```

Codex CLI 會分析錯誤，給出幾種可能的原因和解決方案。我需要理解它的分析，然後選擇合適的方案。這個過程像是和一個技術顧問討論問題。

**3. 崩潰狀態：「我完全不知道該怎麼辦」**

晚上八點，一個複雜的性能問題讓我抓狂。我已經試了各種方法，但問題依舊。

```bash
claude
> 這個頁面加載需要 5 秒，幫我找出所有可能的性能瓶頸並優化
```

這時我需要 Claude Code 接管思考，給我一個完整的解決方案。即使我暫時不理解，至少有了一個能運行的版本，可以慢慢研究。

### 選擇工具的真實邏輯

```
我的狀態判斷流程：

能清楚描述具體步驟嗎？
  ├─ 能 → Aider（我來想，你來寫）
  └─ 不能 → 有參考資料嗎？
      ├─ 有（圖片/文檔）→ Codex CLI（一起探索）

```

## 第二章：真實的使用場景

### 場景一：修 Bug 的心路歷程

**9:00 AM - 發現問題**

用戶回報：「上傳大檔案時會失敗」

**9:15 AM - 初步診斷（Aider）**

```bash
aider src/upload/
> 檢查檔案上傳邏輯，可能是檔案大小限制問題
```

Aider 幫我快速定位相關代碼，但沒找到明顯問題。

**9:30 AM - 深入分析（Codex CLI）**

```bash
# 截圖錯誤訊息
codex --attach network-error.png
> 分析這個網路錯誤，可能的原因是什麼？
```

Codex 指出可能是 CORS 或 timeout 問題。這給了我新的調查方向。

**10:00 AM - 尋求突破（Claude Code）**

```bash
claude
> 這是一個檔案上傳功能，大檔案會失敗。已排除檔案大小限制。
> 請分析所有可能原因並提供完整的解決方案。
```

Claude Code 不僅找到了問題（Nginx 的 `client_max_body_size`），還提供了前後端的完整解決方案，包括分片上傳的實現。

**反思：為什麼需要三個工具？**

- Aider 幫我快速探索已知領域
- Codex CLI 在我有線索時提供分析
- Claude Code 在我束手無策時提供全新視角

### 場景二：從零開始的掙扎

最近要做一個資料視覺化功能，我對 D3.js 一竅不通。

**第一天：讓 Claude 先跑起來**

```bash
claude
> 用 D3.js 實現一個互動式的銷售數據儀表板
> 要有折線圖、餅圖和數據表格
```

Claude 生成了一個完整的解決方案。說實話，我看不懂那些 `selection`、`scale`、`axis` 是什麼，但它能跑！

**第二天：用 Codex 理解細節**

```bash

```

通過 Codex 的解釋，我開始理解 D3 的設計理念。

**第三天：用 Aider 客製化**

```bash
aider dashboard.js
> 修改圖表顏色為公司品牌色
> 添加數據點的 hover 效果
```

現在我已經理解了基本概念，可以用 Aider 進行精確的修改。

## 第三章：工具的真實成本

### 錢的問題

讓我們誠實一點：

- **Aider + DeepSeek**：一天 $5-10，像喝咖啡
- **Aider + Claude 3.7**：一天 $20-50，像吃午餐  
- **Claude Code 重度使用**：一天 $100+，像吃大餐

我的策略：
1. 日常用 DeepSeek（便宜）
2. 複雜問題用 Claude 3.7（值得）
3. 救急時用 Claude Code（必要投資）

### 時間的問題

更重要的是時間成本：

**沒有 AI 時**：
- 研究新技術：2-3 天
- 實現功能：1-2 天  
- 調試優化：1-2 天
- 總計：5-7 天

**有 AI 後**：
- Claude 生成初版：30 分鐘
- 理解和調整：1 天
- 優化完善：1 天
- 總計：2-3 天

省下的時間可以思考更重要的問題。

## 第四章：踩過的坑

### 坑一：盲目信任 AI

曾經，我讓 Claude 生成了一個「完美」的認證系統。部署後才發現，它在 JWT token 裡存了敏感信息。

**教訓**：AI 生成的代碼一定要理解後再用，特別是安全相關的。

### 坑二：過度優化

有次我讓 AI 不斷優化一段代碼，結果從 20 行變成了 200 行的「高度優化」版本。

**教訓**：夠用就好，可讀性比微小的性能提升重要。

### 坑三：忽視上下文

```bash
# 錯誤示範
aider huge-file.js
> 優化所有函數

# 正確做法
aider huge-file.js
> 只優化 calculatePrice 函數，保持其他部分不變
```

**教訓**：給 AI 明確的範圍，避免無謂的大改動。

## 第五章：個人的使用哲學

### 1. AI 是思維的延伸，不是替代

當我用 Claude Code 時，我不是在偷懶，而是在借用它的「思維帶寬」來突破自己的局限。看到 AI 的解決方案後，我常常會想：「原來還可以這樣！」

### 2. 理解比使用更重要

```bash
# 我的學習流程
claude > 生成解決方案
codex > 解釋關鍵部分  
aider > 自己動手改進
```

這個過程讓我真正掌握了知識，而不只是複製貼上。

### 3. 保持懷疑和驗證

AI 很強大，但不是全知的。我養成了這些習慣：
- 關鍵邏輯自己寫測試
- 安全相關代碼必須 review
- 性能相關要實際測量

## 第六章：一個月後的改變

使用這些工具一個月後，我發現自己的工作方式完全改變了：

**以前**：
- 早上：查文檔（2小時）
- 下午：寫代碼（3小時）
- 晚上：調試（2小時）

**現在**：
- 早上：和 AI 討論方案（30分鐘）
- 上午：快速實現（2小時）
- 下午：理解和優化（2小時）
- 多出的時間：思考架構、學習新技術、寫文檔

最大的改變不是寫代碼更快了，而是**有更多時間思考真正重要的問題**。

## 結語：找到你的節奏

每個人使用 AI 工具的方式都不同。有人喜歡讓 AI 主導，有人堅持自己控制。沒有對錯，只有適合不適合。

我的建議：

1. **從小處開始**：先用 AI 改個 bug，寫個函數
2. **保持好奇**：看看 AI 的解法，想想為什麼
3. **建立習慣**：固定在某些場景使用某個工具
4. **定期反思**：這個工具真的幫到我了嗎？

記住，AI 工具的目的不是讓你變成「不會編程的人」，而是讓你變成「更好的程序員」。

最後分享一個小故事：昨天晚上，我用 Claude Code 解決了一個困擾我三天的問題。看著運行成功的程序，我沒有感到挫敗，反而很興奮——因為我又學到了新東西。

這就是 AI 時代的編程：**不是人與機器的競爭，而是人與機器的共舞。**
