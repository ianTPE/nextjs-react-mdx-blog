# AI 編程工具：讓你去到思維無法到達的地方

> 「aider 是人想的比較多，codex 是人要去理解，claude 是沒有辦法去想和理解的時候，讓 claude 先出結果，再事後理解。」—— 一位開發者的真實感悟

就像衣服讓人類能夠生存在極地，AI 工具讓開發者能夠應對超出認知能力的複雜度。這不是人機協作，而是人類能力的延伸。

## 序：從一個下午的掙扎說起

下午三點，我盯著螢幕上的程式碼發呆。一個看似簡單的需求——「將這個 React Class Component 重構成 Hooks」——卻讓我陷入了思維泥潭。不是不會寫，而是有太多種寫法，每種都有利弊，我在各種方案間來回糾結。

這時我打開了 Claude Code：

```bash
claude
> 把這個複雜的 Class Component 重構成 Hooks，保持所有功能不變
```

15 秒後，一個完整的解決方案出現了。我看著生成的代碼，突然明白了——原來可以用 `useReducer` 來管理這些複雜狀態。

**這一刻我意識到：AI 不是在替我思考，而是帶我去了一個我獨自到不了的地方。**

就像登山者需要氧氣罩才能登上珠峰，我需要 AI 才能應對某些複雜度。這不是軟弱，而是明智——知道什麼時候需要工具，並且善用它。

## 第一章：認知負擔與工具選擇

### 人腦的三種狀態

在實際開發中，我發現自己經常處於三種認知狀態，就像面對不同的「天氣」：

**1. 清醒狀態：「我知道要做什麼」（晴天）**

早上剛開始工作，需求明確，思路清晰。這時我會用 Aider——最輕便的裝備：

```bash
aider src/utils/validation.js
> 添加 email 和 phone 的驗證函數，使用正則表達式
```

Aider 就像一件合身的工作服，不會妨礙我的動作，反而讓我做事更俐落。

**2. 迷糊狀態：「我大概知道，但不確定」（起霧）**

下午，遇到一個錯誤訊息：`TypeError: Cannot read property 'map' of undefined`

```bash
codex --attach error-screenshot.png
> 解釋這個錯誤並提供修復方案
```

Codex CLI 就像帶探照燈的外套，幫我在迷霧中看清方向。它不會替我決定往哪走，但會照亮可能的路徑。

**3. 崩潰狀態：「我完全不知道該怎麼辦」（暴風雪）**

晚上八點，一個複雜的性能問題讓我抓狂。這時需要最強力的裝備：

```bash
claude
> 這個頁面加載需要 5 秒，幫我找出所有可能的性能瓶頸並優化
```

Claude Code 就像極地求生裝備——當環境超出人類承受範圍時，它能讓你活下來，甚至找到出路。

### 選擇工具的真實邏輯

就像根據天氣選擇衣服，我根據「認知天氣」選擇工具：

```
今天的編程天氣如何？

思路清晰（晴天）
  └─ Aider（輕便 T-shirt）

有點迷糊（陰天）  
  └─ Codex CLI（防風外套）

完全沒頭緒（暴風雪）
  └─ Claude Code（極地生存裝備）
```

重點是：穿什麼衣服，去的還是你。用什麼工具，編程的還是你。

## 第二章：真實的使用場景

### 場景一：修 Bug 的心路歷程

**9:00 AM - 發現問題**

用戶回報：「上傳大檔案時會失敗」

**9:15 AM - 初步診斷（Aider）**

```bash
aider src/upload/
> 檢查檔案上傳邏輯，可能是檔案大小限制問題
```

Aider 幫我快速定位相關代碼，但沒找到明顯問題。

**9:30 AM - 深入分析（Codex CLI）**

```bash
# 截圖錯誤訊息
codex --attach network-error.png
> 分析這個網路錯誤，可能的原因是什麼？
```

Codex 指出可能是 CORS 或 timeout 問題。這給了我新的調查方向。

**10:00 AM - 尋求突破（Claude Code）**

```bash
claude
> 這是一個檔案上傳功能，大檔案會失敗。已排除檔案大小限制。
> 請分析所有可能原因並提供完整的解決方案。
```

Claude Code 不僅找到了問題（Nginx 的 `client_max_body_size`），還提供了前後端的完整解決方案，包括分片上傳的實現。

**反思：為什麼需要三個工具？**

- Aider 幫我快速探索已知領域
- Codex CLI 在我有線索時提供分析
- Claude Code 在我束手無策時提供全新視角

### 場景二：從零開始的掙扎

最近要做一個資料視覺化功能，我對 D3.js 一竅不通。

**第一天：讓 Claude 先跑起來**

```bash
claude
> 用 D3.js 實現一個互動式的銷售數據儀表板
> 要有折線圖、餅圖和數據表格
```

Claude 生成了一個完整的解決方案。說實話，我看不懂那些 `selection`、`scale`、`axis` 是什麼，但它能跑！

**第二天：用 Codex 理解細節**

```bash
codex
> 解釋這段 D3.js 代碼中的 scale 和 axis 是如何工作的
```

通過 Codex 的解釋，我開始理解 D3 的設計理念。

**第三天：用 Aider 客製化**

```bash
aider dashboard.js
> 修改圖表顏色為公司品牌色
> 添加數據點的 hover 效果
```

現在我已經理解了基本概念，可以用 Aider 進行精確的修改。

### 場景三：重構的猶豫與決斷

面對一個 2000 行的「神類」，我知道要重構，但不知從何下手。

```bash
# 先讓 Claude 分析

> 分析 UserService.js，提出重構方案

# Claude 建議拆分成 5 個模組後
aider UserService.js

# 遇到複雜的依賴關係時

> 這些方法之間的依賴關係圖是什麼樣的？
```

## 第三章：工具的真實成本

### 錢的問題

讓我們誠實一點：

- **Aider + DeepSeek**：一天 $5-10，像喝咖啡
- **Aider + Claude 3.7**：一天 $20-50，像吃午餐  
- **Claude Code 重度使用**：一天 $100+，像吃大餐

我的策略：
1. 日常用 DeepSeek（便宜）
2. 複雜問題用 Claude 3.7（值得）
3. 救急時用 Claude Code（必要投資）

### 時間的問題

更重要的是時間成本：

**沒有 AI 時**：
- 研究新技術：2-3 天
- 實現功能：1-2 天  
- 調試優化：1-2 天
- 總計：5-7 天

**有 AI 後**：
- Claude 生成初版：30 分鐘
- 理解和調整：1 天
- 優化完善：1 天
- 總計：2-3 天

省下的時間可以思考更重要的問題。

## 第四章：踩過的坑

### 坑一：盲目信任 AI

曾經，我讓 Claude 生成了一個「完美」的認證系統。部署後才發現，它在 JWT token 裡存了敏感信息。

**教訓**：AI 生成的代碼一定要理解後再用，特別是安全相關的。

### 坑二：過度優化

有次我讓 AI 不斷優化一段代碼，結果從 20 行變成了 200 行的「高度優化」版本。

**教訓**：夠用就好，可讀性比微小的性能提升重要。

### 坑三：忽視上下文

```bash
# 錯誤示範
aider huge-file.js
> 優化所有函數

# 正確做法
aider huge-file.js
> 只優化 calculatePrice 函數，保持其他部分不變
```

**教訓**：給 AI 明確的範圍，避免無謂的大改動。

## 第五章：個人的使用哲學

### 1. AI 是裝備，不是替身

就像登山者不會說「是氧氣罩征服了珠峰」，我也不會說「是 AI 寫了這個系統」。AI 只是讓我能在「複雜度稀薄」的高處生存的裝備。

### 2. 理解比使用更重要

```bash
# 我的學習流程
claude > 生成解決方案（到達新地方）
codex > 解釋關鍵部分（熟悉地形）  
aider > 自己動手改進（建立據點）
```

就像探險家會研究新大陸的地形，我會研究 AI 帶我到達的新領域。

### 3. 保持對「極限」的認知

再好的防寒服也有溫度極限，再強的 AI 也有能力邊界：
- 關鍵邏輯自己寫測試（AI 可能忽略邊界情況）
- 安全相關代碼必須 review（AI 沒有真實世界的風險意識）
- 性能相關要實際測量（AI 的「優化」可能是過度設計）

## 第六章：一個月後的改變

使用這些工具一個月後，我發現自己的工作方式完全改變了：

**以前**：
- 早上：查文檔（2小時）
- 下午：寫代碼（3小時）
- 晚上：調試（2小時）

**現在**：
- 早上：和 AI 討論方案（30分鐘）
- 上午：快速實現（2小時）
- 下午：理解和優化（2小時）
- 多出的時間：思考架構、學習新技術、寫文檔

最大的改變不是寫代碼更快了，而是**有更多時間思考真正重要的問題**。

## 第七章：能力邊界的擴展

使用 AI 工具後，我發現自己的「可達區域」擴大了：

**技術棧的邊界**
- 以前：精通 2-3 個框架，其他不敢碰
- 現在：可以快速上手任何主流技術

上週，我用 Rust 寫了一個性能關鍵的模組。我不會 Rust，但 Claude Code 幫我寫出了能跑的版本，然後我通過 Codex 理解了所有權系統，最後用 Aider 優化了性能。一週時間，從零到生產可用。

**複雜度的邊界**  
- 以前：超過 1000 行的重構就頭疼
- 現在：敢於挑戰整個系統的架構升級

**創新的邊界**
- 以前：想法受限於已知的解決方案
- 現在：可以探索多種可能性

最近在設計一個實時協作系統時，我讓 Claude 給出 5 種不同的架構方案。其中有兩種是我從沒想過的——使用 CRDT 和 Event Sourcing 的組合。這開啟了全新的思路。

但要記住：**工具擴展了邊界，但行走的還是你自己。**

## 結語：AI 是衣服，不是舞伴

有人說 AI 時代是「人機共舞」，但我更願意把 AI 比作衣服。

人類發明衣服，不是為了和衣服跳舞，而是為了去更冷的地方、更熱的地方、更危險的地方。衣服讓我們的活動範圍從溫帶擴展到極地，從陸地擴展到海洋，從地球擴展到太空。

AI 工具也是一樣：

- **Aider** 像是日常的襯衫，輕便舒適，適合日常穿著
- **Codex CLI** 像是多功能外套，口袋很多，適應各種場景  
- **Claude Code** 像是極地防寒服，讓你進入思維的無人區

穿上這些「衣服」後，我發現自己能去以前去不了的地方：

**以前不敢碰的技術棧**，現在可以大膽嘗試。不懂 Rust？沒關係，Claude Code 先寫個能跑的版本，我再慢慢理解。

**以前想不出的解決方案**，現在有了可能。面對複雜的分布式系統問題，我可以讓 AI 提供五種不同的架構方案。

**以前沒時間做的優化**，現在變得可行。代碼重構、性能調優、測試覆蓋，這些「重要但不緊急」的事情，終於有時間做了。

但請記住：穿上衣服的還是你自己。

衣服不會替你走路，不會替你思考，不會替你做決定。它只是讓你在嚴酷的環境中生存下來，讓你走得更遠。

同樣，AI 不會替你編程，不會替你設計架構，不會替你理解業務。它只是讓你在複雜度爆炸的現代軟體開發中，依然能夠從容應對。

**最重要的是**：不要因為有了保暖的衣服就忘記鍛煉身體。不要因為有了 AI 就放棄學習和思考。

昨天，我用 Claude Code 解決了一個困擾三天的問題。今天，我花了兩個小時研究它的解決方案，學到了新的設計模式。這就像登山者研究新裝備的原理——不是為了依賴它，而是為了更好地使用它，去攀登更高的山峰。
